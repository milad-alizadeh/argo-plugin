# Design memory-ownership model

A small set of artifacts holds design-pack state. Each fact has exactly ONE owner; every
other artifact that needs it either points at the owner or is silent about
it — no fact has two owners, and no artifact is a hand-mirrored copy of
another's data (design-memory-placement.md, council synthesis).

| Fact | Owner | Notes |
|---|---|---|
| Identity (component name + `nodeId`) | `design/registry.json` | Thin pointer index, healed on read (`getNodeByIdAsync` miss → `findAll` by name within the category frame → persist the corrected id), verified before use — never trusted blind. |
| Geometry / variants / sizing | `design/specs/<Component>.json` | One `get_metadata` call on a known `nodeId` returns variant props; never copied into the registry. |
| Code mapping (`storyId`, `importPath`, `propMapping`) | `design/story-map.json` | Regenerated by `figma-sync` — file-derived, cannot drift. The registry does NOT derive story-map; the dependency runs `file → figma-sync → story-map → registry`, never the reverse. |
| Intent / rulings (mood, DO/DON'T list) | `aesthetic-profile.md` | Auto-discovered, read before hi-fi work. A per-component justification for departing from a ruling goes in the registry's one-line `description` or an audit waiver — never a `rulingsApplied[]` pointer array (rejected: an independent-edit sync liability that doesn't solve the underlying drift). |
| Audit pass/fail | `design/registry.json`'s `status` (`audit-clean` = last self-audit passed) | NOT a pointer into `design/audit-receipt.json` — that file is a single-slot, batch-scoped last-run record overwritten every audit, so a per-component pointer into it is stale immediately. `provenance`'s inline `lastAudit` timestamp/task-name detail was dropped in design-system-reset-overhaul.md Slice 4 (not in the design doc's 5-field registry shape) — `status` alone carries the pass/fail signal now, with no per-component audit timestamp. |
| Semantic token vocabulary | `design/semantic-manifest.md` | Generated by `argo design generate-token-manifest` from `design/tokens.json` at `figma-sync` time (~60 semantic token names + purposes) — the authoring vocabulary LLM sessions bind against, never hand-edited, never an enumeration of the file's ~1800 local primitives. |

## What's deliberately absent (and why)

- **`variants[]` in the registry** — derivable from specs / one live
  `get_metadata` call; a JSON copy only drifts.
- **`kitDeps[]` in the registry** — no longer applicable (the base-congruence
  walker it was derivable from is retired); only mattered at `design-upgrade`
  scale, added there if ever needed.
- **`rulingsApplied[]` pointer array** — a maintenance liability against an
  independently-edited `aesthetic-profile.md` that doesn't solve chat-only
  ruling drift (already fixed by aesthetic-profile auto-discovery).
- **`lastAuditReceipt` pointer** — dangles into a single-slot batch file
  overwritten every run; replaced by the inline `lastAudit` snapshot.
- **`synced`/`coded` status values** — derived (story-map presence ⇒ synced;
  a passing spec-diff receipt ⇒ coded), never stored; storing them
  guarantees 3-way drift against the files that actually decide them.
- **Status in the component's Figma description** — status transitions fire
  from skills that don't write to Figma (e.g. `figma-to-code`'s `coded`
  flip), so an in-file copy can't be kept in step, and forcing it would
  manufacture a spurious `use_figma` write (bumping `design-guard-record.mjs`'s
  writeCount, which then owes a fresh audit receipt) for no reader that
  currently exists.

## Enforcement

`RegistryEntrySchema` (`packages/figma-design-kit/schemas.js`) is the
schema-enforced join contract: it reuses `StoryMapEntrySchema`'s `nodeId`
field name/type, so the registry↔story-map join key type-matches by
construction — a future skill author adding a new registry field must extend
the schema, which is the natural place to notice a fact already owned
elsewhere.
